## 一、认识 Pi：终端里那个"不务正业"的 AI 助手

Pi（官网：https://pi.dev）是一个运行在终端里的 AI 编码助手。它由 Mario Zechner（知名开源游戏引擎 libGDX 的作者）创建，目前 GitHub 上已有超过 44,000 个 Star。

但 Pi 真正让人惊讶的不是它的知名度，而是它的设计思路。

它的核心 Agent Loop 只有 **418 行 TypeScript 代码**。

对比一下：Claude Code 的核心循环上千行，LangChain 的 Agent 框架更是数千行起步。而 Pi 用 418 行就实现了完整的消息转换、LLM 调用、工具执行、事件流——而且性能更好。

这是怎么做到的？

---

## 二、418 行代码背后的设计哲学

### 2.1 双循环架构：应用层和模型层彻底分离

Pi 的核心是一个精巧的双循环架构：

```
用户输入 → [应用层消息循环] → LLM 调用 → [模型层消息循环] → 工具执行 → 回到应用层
```

**应用层**负责管理完整的会话上下文、工具调用结果、用户交互。**模型层**只关心发送给 LLM 的 token 序列。

这种分离带来两个关键好处：

1. **精确控制上下文**：你可以决定哪些历史消息发送给模型，哪些保留在本地。这意味着长会话不会爆上下文窗口——Pi 会自动压缩旧消息，但完整历史依然保留在会话文件中。
2. **事件驱动的扩展系统**：每个环节都暴露了事件钩子，扩展可以拦截消息、修改工具调用、注入上下文——而不需要修改核心代码。

### 2.2 核心只做四件事

Pi 的核心只提供四个工具：

| 工具 | 用途 |
| --- | --- |
| `read` | 读取文件内容 |
| `write` | 创建或覆盖文件 |
| `edit` | 精确编辑文件（基于字符串匹配） |
| `bash` | 执行 Shell 命令 |

就这四个。没有子代理，没有计划模式，没有 MCP，没有权限弹窗。

但这恰恰是 Pi 最聪明的地方。

### 2.3 扩展系统：一切皆可插拔

Pi 的扩展系统通过 `jiti` 在运行时加载 TypeScript 模块。一个扩展可以做到：

- **注册自定义工具**：让 LLM 调用任何 API
- **订阅 10+ 个生命周期事件**：拦截消息、工具调用、上下文变化
- **创建自定义 UI 组件**：在终端里渲染交互式界面
- **注册自定义命令**：像 `/mycommand` 一样调用
- **会话状态持久化**：数据跨重启保存

这意味着：子代理、计划模式、MCP 支持、权限控制——这些其他工具内置的功能，在 Pi 里全部由扩展实现。你想用什么就装什么，不想用就不装。

用 Pi 官方文档的话说：\*\*"不内置 MCP。不内置子代理。不内置计划模式。"\*\*

这种设计让 Pi 的 npm 生态快速爆发。目前已有超过 **2,100 个** npm 包标注为 pi-package，涵盖网络访问、代码导航、浏览器操控、多会话协作等方方面面。

**甚至有意思的是, 你可以自己让pi给自己写拓展.**

## 三、13 个核心插件详解：从基础到进阶

下面按照功能类别，逐一拆解 13 个最实用的 Pi 插件。

---

### 🔧 第一类：核心能力扩展

#### 1. pi-web-access — 给 Pi 联网

**它解决什么问题**：Pi 默认不能上网。这个插件赋予它搜索网页、抓取内容、提取 YouTube 字幕的能力。

**核心技术细节**：

- 支持多搜索引擎后端：Exa、Perplexity、Gemini、Brave、DuckDuckGo
- 零配置即可使用（无需 API Key 的基础搜索）
- 网页自动转 Markdown，方便 LLM 理解
- 注册 `web_search`、`fetch_content`、`get_search_content` 三个工具

**典型使用场景**：

- 写代码时搜索最新 API 文档
- 做技术调研时抓取和对比多个信息源
- 分析 GitHub Issue 页面内容

**安装**：`pi install npm:pi-web-access`

---

#### 2. pi-mcp-adapter — 连接 MCP 生态

**它解决什么问题**：Pi 不内置 MCP 支持，但 MCP 生态有大量实用工具（数据库、浏览器、文件系统等）。这个插件让 Pi 能无缝接入。

**核心技术细节**：

- 注册一个约 200 token 的代理工具（不是加载全部工具定义）
- 按需发现和调用 MCP 服务器工具
- 支持 stdio、SSE、StreamableHTTP、WebSocket 四种传输协议
- 工具元数据本地缓存，支持离线搜索
- 支持从 Claude Desktop、Cursor 等工具导入 MCP 配置

**为什么它很聪明**：传统 MCP 集成会把所有工具定义加载到上下文中，消耗大量 token。pi-mcp-adapter 只用一个代理工具，按需加载——上下文占用极小。

**典型使用场景**：

- 连接数据库 MCP 服务器，让 Pi 直接查询数据
- 连接浏览器 MCP，做 Web 自动化
- 连接文件系统 MCP，管理远程文件

**安装**：`pi install npm:pi-mcp-adapter`

---

#### 3. @tintinweb/pi-subagents — 子代理管理

**它解决什么问题**：复杂任务需要分工协作。这个插件让 Pi 能启动多个子代理并行工作。

**核心技术细节**：

- 每个子代理拥有独立会话、独立工具集、独立模型配置
- 支持前台和后台执行模式
- 支持中途 steering（引导子代理方向）
- 实时 Widget UI 显示子代理状态
- 多种代理类型：Explore（代码探索）、general-purpose（通用任务）
- 支持自定义代理定义（Markdown 文件配置）

**典型使用场景**：

- 重构项目时，同时启动 3 个子代理：代码分析、最佳实践搜索、测试编写
- 研究新库时，并行读取文档、源码和社区讨论
- 主代理汇总所有子代理的结果

**安装**：`pi install npm:@tintinweb/pi-subagents`

---

#### 4. pi-agents-switch — 代理角色切换

**它解决什么问题**：不同任务需要不同的"人设"。写代码和做规划，需要的系统提示和工具权限完全不同。

**核心技术细节**：

- 用 YAML 前导码定义代理：名称、描述、模型、工具集、思考级别
- `/switch` 命令一键切换
- `Ctrl+Shift+M` 循环切换，`Alt+S` 搜索代理
- 支持继承默认配置，只覆盖差异项
- 代理可限制工具权限（例如禁止 Planner 使用 `edit` 工具）

**典型使用场景**：

- Planner 代理：只规划不写代码（禁止 edit 工具）
- Writer 代理：专注写作，配置专门的写作工作流
- Scholar 代理：学术研究，配置深度推理
- 不同任务切换不同角色，每个都有最优配置

**安装**：`pi install npm:pi-agents-switch`

### 🧠 第二类：交互与智能增强

#### 5. @juicesharp/rpiv-ask-user-question — 结构化提问

**它解决什么问题**：LLM 经常"猜测"用户意图，猜错了要返工。这个插件让 Pi 在需要决策时弹出结构化问卷。

**核心技术细节**：

- 支持单选和多选问题（每次最多 4 个）
- 每个选项有标签和详细描述
- 单选问题支持左右分栏预览对比
- 问题卡片式 UI，比纯文本清晰得多

**典型使用场景**：

- 选技术方案时，弹出 A/B/C 三个选项让你选
- 确定项目结构时，列出多种目录组织方式
- 配置参数时，提供数值选项

**安装**：`pi install npm:@juicesharp/rpiv-ask-user-question`

---

#### 6. @juicesharp/rpiv-todo — 任务追踪

**它解决什么问题**：多步骤任务容易"迷路"——Pi 做着做着忘了还要做什么。

**核心技术细节**：

- 任务 4 种状态：pending → in\_progress → completed / deleted
- 支持任务依赖关系（blockedBy）
- 跨会话持久化（重启 Pi 后任务列表还在）
- 自动标记当前任务状态

**典型使用场景**：

- 多步骤功能开发时，自动创建 todo 追踪进度
- 复杂重构时，拆成小任务逐个完成
- 即使中途退出，恢复会话后任务列表依然完整

**安装**：`pi install npm:@juicesharp/rpiv-todo`

---

#### 7. @juicesharp/rpiv-btw — 旁路提问

**它解决什么问题**：Pi 正在执行长任务时，你想问个不相关的问题，但不想污染主会话上下文。

**核心技术细节**：

- 创建只读副本会话回答旁路问题
- 主会话上下文完全不受影响
- 支持在 Pi 执行工具期间发送 steering 消息

**典型使用场景**：

- Pi 在写代码时，临时查一个 API 用法
- 等 Pi 跑长任务时，顺手让它搜索资料

**安装**：`pi install npm:@juicesharp/rpiv-btw`

### 🎨 第三类：体验与效率增强

#### 8. pi-hud — 抬头显示器

**它解决什么问题**：你不知道 Pi 消耗了多少 token、花了多少钱、上下文用了多少。

**核心技术细节**：

- 实时显示：模型名称、token 用量（输入/输出/缓存读写）、成本、上下文使用率
- 显示项目路径、Git 分支、子代理活动状态
- 支持顶部/底部、左/右多种位置
- `Ctrl+Shift+H` 切换显示，`Ctrl+.` 切换模式
- 支持自动压缩模式（流式输出时自动缩小）

**典型使用场景**：

- 随时监控 token 消耗，避免意外超支
- 长任务时关注上下文使用率，快满了就 `/compact`
- 对比不同模型的成本效率

**安装**：`pi install npm:pi-hud`

---

#### 9. pi-rewind — 时光倒流

**它解决什么问题**：AI 改代码难免出错，手动 git reset 很麻烦。

**核心技术细节**：

- 每次 edit/write/bash 后自动创建 Git checkpoint
- 智能去重（工作树没变化不重复创建）
- `/rewind` 回退前展示 diff 预览
- 支持多级 undo/redo
- checkpoint 以 Git ref 形式存储，跨重启持久化
- 自动清理旧会话的 checkpoint
- 安全排除大文件和敏感文件

**典型使用场景**：

- Pi 改错代码，一键回退到修改前
- 实验性重构前有"后悔药"
- 多步骤操作后回到任意中间状态

**安装**：`pi install npm:pi-rewind`

### 🔗 第四类：外部集成

#### 10. pi-intercom — 多会话通信

**它解决什么问题**：多个 Pi 会话需要协作，但不能互相通信。

**核心技术细节**：

- 基于本地 Unix Socket Broker 的 1 对 1 通信
- 按会话名称或 ID 精确路由消息
- 支持发送上下文、查询结果、任务请求
- 收到消息自动触发新一轮对话
- 小型 overlay UI 显示通信状态

**典型使用场景**：

- Planner-Worker 模式：一个会话规划，另一个执行
- 多项目协作：跨项目传递上下文
- 复杂任务分工：拆成子任务分发给不同会话

**安装**：`pi install npm:pi-intercom`

---

#### 11. pi-chrome — 浏览器操控

**它解决什么问题**：很多 Web 操作需要登录态，无头浏览器做不到。

**核心技术细节**：

- 连接到已登录的 Chrome 配置文件（需显式授权 `/chrome authorize`）
- 标签页管理：打开、关闭、切换
- 页面交互：点击、输入、填表
- 页面检查：截图、DOM 评估
- 基于本地回环连接（127.0.0.1:17318），拒绝外部请求
- MIT 许可的轻量 Chrome 扩展

**典型使用场景**：

- 自动在 GitHub 创建 Issue，填表单
- 登录内部系统自动抓取数据
- Web 自动化测试

**安装**：`pi install npm:pi-chrome`

---

#### 12. pi-wechat-assistant — 微信助手

**它解决什么问题**：不在电脑前时，无法跟 Pi 交互。

**核心技术细节**：

- 连接微信 iLink Bot
- 二维码扫码登录
- 长轮询接收微信消息，注入 Pi 会话
- Pi 回复后自动发回微信
- 支持自动启动

**典型使用场景**：

- 通过微信给 Pi 下达任务
- Pi 完成工作后微信通知
- 随时随地跟 AI 助手对话

**安装**：`pi install npm:pi-wechat-assistant`

### 🔬 第五类：开发者专属

#### 13. pi-lens — 代码导航增强

**它解决什么问题**：Pi 默认只能文本搜索代码，无法理解代码结构。

**核心技术细节**：

- **ast-grep 集成**：基于 AST 的结构化搜索，比正则匹配精准得多
- **LSP 导航**：跳转定义、查找引用、重命名符号
- **诊断信息**：实时 lint 错误、类型错误、警告
- **安全分析**：代码漏洞检测
- **支持 36+ 语言**：TypeScript、Python、Rust、Go、Java、C/C++ 等
- 附带 4 个技能：ast-grep 搜索、LSP 导航、编写 ast-grep 规则、编写 tree-sitter 查询规则

**典型使用场景**：

- 重构时用 ast-grep 找到所有需要修改的调用点
- 理解陌生代码时用 LSP 跳转定义、查看引用链
- 提交前跑一遍诊断，确保没有类型错误

**安装**：`pi install npm:pi-lens`

## 五、快速上手指南

### 安装 Pi

```
# 方式一：npm 安装
npm install -g @earendil-works/pi-coding-agent

# 方式二：一键安装脚本
curl -fsSL https://pi.dev/install.sh | sh
```

### 配置 API Key

```
# 以 Anthropic 为例
export ANTHROPIC_API_KEY=sk-ant-...

# 启动
pi
```

### 安装插件

```
# 按需安装
pi install npm:pi-web-access
pi install npm:pi-rewind
pi install npm:@juicesharp/rpiv-todo
pi install npm:pi-lens
```

### 常用命令

| 命令 | 功能 |
| --- | --- |
| `/model` | 切换模型 |
| `/settings` | 配置参数 |
| `/resume` | 恢复历史会话 |
| `/compact` | 压缩上下文 |
| `/fork` | 分支会话 |
| `/rewind` | 回退操作 |
| `/switch` | 切换代理 |
| `/hud` | 切换抬头显示 |
| `/reload` | 重载扩展和配置 |

## 六、总结

Pi Agent 的设计思路代表了一种趋势：\*\*在 AI 工具领域，最好的架构不是"大而全"，而是"小而美 + 可生长"\*\*。

它的 418 行核心循环 + 2000+ 个社区扩展，证明了一件事：当一个工具把扩展性做到极致，用户自己会把它变成任何他们需要的样子。

如果你受够了 GUI 工具的臃肿，或者想完全掌控 AI 助手的行为，Pi 值得一试。
